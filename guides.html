Learning Guides

Suricata 기반 보안 로그를 수집하고, CSV 데이터셋으로 변환한 뒤, 머신러닝 기반 IDS 실습까지 진행하는 전체 흐름을 단계별로 정리한 가이드입니다.

1. Suricata 로그 수집

실습의 시작은 Suricata의 eve.json 로그 수집입니다. Suricata는 alert, flow, http, dns, tls, fileinfo 등 다양한 이벤트를 JSON 형식으로 저장할 수 있으며, 그중 교육 및 머신러닝 실습에서는 주로 flow 이벤트와 alert 이벤트를 중심으로 사용합니다.

일반적으로 flow 이벤트는 패킷 수, 바이트 수, 시작 시각, 종료 시각과 같은 통계 정보를 포함하고 있으며, alert 이벤트는 특정 flow에 대해 어떤 탐지 규칙이 동작했는지를 보여줍니다. 따라서 flow_id를 기준으로 두 이벤트를 연결하면, 네트워크 흐름 단위의 학습용 데이터셋을 구성할 수 있습니다.

실제 실습에서는 Ubuntu 또는 T-Pot 환경에서 Suricata를 구동한 뒤, 저장된 eve.json을 별도로 백업하고, 실험 날짜와 수집 조건을 함께 기록해두는 것이 좋습니다. 이렇게 해야 나중에 데이터 출처와 실험 조건을 재현하기 쉽습니다.

2. CSV 변환 및 라벨링

수집된 eve.json은 바로 머신러닝에 넣기 어렵기 때문에, 필요한 필드만 추출하여 CSV 형태로 변환해야 합니다. 이때 자주 사용하는 기본 필드는 proto, src_port, dest_port, pkts_toserver, pkts_toclient, bytes_toserver, bytes_toclient 등입니다.

라벨링은 교육 목적상 단순하고 명확해야 합니다. 본 사이트의 기본 예시는 BENIGN=0, ATTACK=1 기준을 사용합니다. 즉, flow_id와 연결된 alert가 존재하면 공격으로 간주하고 1, 그렇지 않으면 정상으로 간주하고 0으로 라벨링합니다.

이 과정에서 중요한 점은 단순히 event_type이 alert인 로그만 공격으로 보는 것이 아니라, flow 단위에서 alert 발생 여부를 매핑하는 것입니다. 그래야 flow 기반 특징과 라벨이 일관되게 연결됩니다.

3. 데이터 전처리 및 결측치 처리

CSV로 정리한 뒤에는 바로 학습하지 말고 먼저 전처리를 수행해야 합니다. 우선 컬럼명의 공백을 제거하고, 숫자형이어야 하는 필드를 숫자형으로 변환합니다. 그 다음 결측치와 무한대 값을 확인하여 제거하거나 적절한 기본값으로 보정합니다.

categorical 필드인 proto, app_proto, flow_state는 문자열 형태일 수 있으므로, 모델 학습 전에는 Label Encoding과 같은 방식으로 숫자화해야 합니다. 다만 학습 데이터에 없던 새로운 값이 예측 단계에서 등장할 수 있으므로, UNKNOWN 처리나 fallback 규칙도 함께 고려하는 것이 좋습니다.

이 단계에서 누수 가능성이 있는 식별 컬럼도 제거해야 합니다. 예를 들어 flow_id 자체나 alert_signature, alert_category 같은 컬럼은 모델이 공격 여부를 너무 쉽게 맞혀버릴 수 있으므로 일반적인 학습 feature에서는 제외하는 것이 바람직합니다.

4. 데이터 밸런싱

실제 보안 로그는 정상 트래픽이 훨씬 많고 공격 트래픽이 적은 경우가 많습니다. 이런 상태로 바로 학습하면 모델이 정상 클래스에 치우쳐 학습될 가능성이 큽니다. 따라서 교육용 실습에서는 클래스 불균형을 어느 정도 조정해주는 것이 좋습니다.

본 연구 흐름에서는 일반적으로 정상 : 공격 = 3 : 1 비율을 사용합니다. 공격 데이터를 모두 사용하고, 정상 데이터는 그 3배만 샘플링하여 학습 데이터셋을 구성하면 과도한 클래스 편향을 줄일 수 있습니다.

단, 밸런싱은 실험 목적에 따라 달라질 수 있습니다. 실제 운영 환경을 최대한 반영하고 싶다면 원본 분포를 유지한 평가도 함께 수행해보는 것이 좋습니다.

5. Feature Engineering

단순 원본 필드만으로는 공격 패턴을 충분히 구분하기 어려울 수 있기 때문에, 파생 feature를 추가로 생성하는 과정이 중요합니다. 예를 들어 total_pkts, total_bytes, duration과 같은 기본 합성 feature를 먼저 만들 수 있습니다.

그 다음 packet_rate, byte_rate를 통해 시간 대비 트래픽 밀도를 계산하고, flow_ratio, pkt_ratio, byte_ratio를 통해 서버 방향과 클라이언트 방향의 비대칭성을 표현할 수 있습니다. 이런 값들은 정상 연결과 비정상 연결의 차이를 더 잘 드러내는 경우가 많습니다.

교육용 실습에서는 파생 feature를 만들 때 어떤 수식으로 계산했는지 함께 문서화하는 것이 중요합니다. 그래야 나중에 CSV만 보더라도 각 컬럼의 의미를 바로 이해할 수 있습니다.

6. MLP 모델 학습

전처리와 Feature Engineering이 완료되면, PyTorch 기반의 MLP 모델로 이진 분류 학습을 수행할 수 있습니다. 일반적으로 입력 feature 수에 맞춰 첫 번째 Linear 레이어를 설정하고, 중간에 ReLU, BatchNorm, Dropout을 조합하여 학습 안정성과 일반화 성능을 확보합니다.

학습 데이터는 Train, Validation, Test로 나누어 사용하며, 입력값은 StandardScaler로 정규화한 뒤 모델에 전달합니다. 클래스 불균형이 남아 있을 경우에는 class weight를 주거나, threshold 최적화 단계에서 precision 중심 전략을 사용하는 방식으로 보완할 수 있습니다.

Ubuntu 환경에서 GPU를 사용할 경우, CUDA가 정상 인식되는지 먼저 확인한 뒤 PyTorch에서 device를 cuda로 설정해야 합니다. 학습이 끝나면 모델 가중치, scaler, feature_columns, categorical encoders를 함께 저장해두어야 예측 단계에서 동일한 기준을 유지할 수 있습니다.

7. 외부 Suricata 로그 예측

모델 학습이 끝난 뒤에는 새로운 Suricata 로그에 대해 예측을 수행할 수 있습니다. 이때 중요한 점은 외부 로그도 학습 때와 같은 feature 구조로 맞춰야 한다는 것입니다. 즉, 학습 시 사용한 feature_columns 순서와 동일하게 정렬하고, 없는 컬럼은 기본값으로 생성해야 합니다.

또한 categorical 인코딩도 반드시 학습 시 저장한 encoder 기준으로 적용해야 합니다. 학습에 없던 값이 들어오면 UNKNOWN 또는 fallback 값으로 처리해야 하며, 숫자형 필드는 scaler를 다시 학습하면 안 되고 학습 때 저장한 scaler를 그대로 사용해야 합니다.

이 과정을 정확히 지키지 않으면 예측 입력 데이터 분포가 달라져 성능이 급격히 떨어질 수 있습니다.

8. Threshold 최적화

이진 분류 모델은 보통 공격 확률을 출력하므로, 어느 값을 기준으로 공격과 정상으로 나눌지 결정하는 threshold 설정이 필요합니다. 기본값 0.5를 사용할 수도 있지만, 보안 환경에서는 false positive를 줄이기 위해 precision 중심으로 threshold를 조정하는 경우가 많습니다.

본 연구 기준에서는 먼저 Precision ≥ 0.90을 만족하는 threshold를 찾고, 그 조건을 만족하는 후보들 중에서 F1-score가 가장 높은 threshold를 선택합니다. 만약 그런 threshold가 전혀 없다면, 그때는 전체 후보 중 F1-score가 가장 높은 값을 fallback으로 사용합니다.

이 전략은 무조건 공격을 많이 잡는 것보다, 실제 운영에서 불필요한 오탐 경고를 줄이는 방향에 더 가깝습니다.

9. 성능 평가와 결과 해석

모델 평가에서는 Accuracy만 보지 말고, Precision, Recall, F1-score를 함께 확인해야 합니다. 특히 보안 탐지에서는 공격을 놓치지 않는지도 중요하고, 정상 트래픽을 과하게 공격으로 판단하지 않는지도 중요합니다.

따라서 Confusion Matrix를 통해 True Positive, False Positive, False Negative, True Negative를 함께 해석해야 합니다. 추가로 ROC Curve와 Precision-Recall Curve를 저장하면 threshold 변화에 따른 성능 변화를 더 쉽게 이해할 수 있습니다.

실습 결과는 날짜와 실행 번호 기준으로 저장해두면 좋습니다. 예를 들어 2026-04-07/run_001 형태로 정리하면 어떤 모델이 어떤 데이터와 threshold로 평가되었는지 추적하기 쉽습니다.

10. 연구 및 공개 시 주의사항

교육용으로 데이터를 공개할 때는 민감한 정보가 포함되지 않도록 주의해야 합니다. 실제 IP, 내부 네트워크 구조, 사용자 정보, 민감한 서비스 식별 정보는 그대로 공개하지 않는 것이 좋습니다.

또한 공개용 CSV는 원본 전체보다 비식별화된 샘플 데이터나 실습용으로 정리된 버전을 우선 제공하는 것이 안전합니다. 데이터셋 설명 문서에는 수집 환경, 라벨 기준, 주요 feature, 전처리 방식, 사용 시 주의사항을 함께 적어두는 것이 바람직합니다.

이렇게 정리하면 이 사이트는 단순한 파일 저장소가 아니라, 보안 입문자와 연구자 모두가 활용할 수 있는 교육용 학습 아카이브로 발전할 수 있습니다.
